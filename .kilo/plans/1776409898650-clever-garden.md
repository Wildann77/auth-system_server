# Google OAuth Integration Plan

## Overview
Integrate Google OAuth 2.0 authentication into the existing Node.js/TypeScript auth system. This will allow users to register and login using their Google accounts alongside the existing local authentication.

## Current State Assessment
- ✅ **Dependencies**: `google-auth-library` already installed
- ✅ **Database Schema**: `Provider` enum includes `GOOGLE`, `User.passwordHash` is nullable
- ✅ **Environment Variables**: Google OAuth variables defined in `.env.example`
- ✅ **Auth Service**: Already handles provider checking for OAuth users
- ✅ **Infrastructure**: Ready for OAuth implementation

## Implementation Plan

### Phase 1: Infrastructure Preparation
**Status**: Mostly Complete

- [x] Verify `google-auth-library` dependency (already installed)
- [x] Confirm Google OAuth environment variables in `.env.example`
- [x] Ensure Prisma schema supports OAuth users (nullable passwordHash, provider field)

**Remaining**:
- [ ] User must configure actual Google OAuth credentials in Google Cloud Console
- [ ] Copy `.env.example` to `.env` and fill in Google OAuth variables

### Phase 2: Schema & Validation Layer
Add new Zod schemas for Google OAuth endpoints.

**Files to modify**:
- `src/features/auth/auth.schema.ts`

**New schemas needed**:
- `googleCallbackSchema`: Validates callback query parameters (code, state)
- Export new input types

### Phase 3: Service Layer Implementation
Add Google OAuth business logic to auth service.

**Files to modify**:
- `src/features/auth/auth.service.ts`

**New methods needed**:
- `getGoogleAuthUrl()`: Generate Google OAuth authorization URL
- `processGoogleCallback(code: string)`: Handle OAuth callback, verify tokens, sync user

**Logic requirements**:
- Exchange authorization code for Google tokens
- Verify Google ID token and extract user info (email, name, picture)
- User synchronization:
  - If user exists: Update `isEmailVerified: true` and generate tokens
  - If user doesn't exist: Auto-register with `provider: GOOGLE`, `isEmailVerified: true`
- Generate local access/refresh tokens following existing hybrid token strategy

### Phase 4: Controller Layer Implementation
Add HTTP handlers for Google OAuth endpoints.

**Files to modify**:
- `src/features/auth/auth.controller.ts`

**New methods needed**:
- `googleStart()`: Redirect to Google OAuth URL
- `googleCallback()`: Process callback, set refresh token cookie, return access token

### Phase 5: Routing Layer
Add new routes for Google OAuth flow.

**Files to modify**:
- `src/features/auth/auth.routes.ts`

**New routes needed**:
- `GET /api/v1/auth/google`: Initiate OAuth flow
- `GET /api/v1/auth/google/callback`: Handle OAuth callback

### Phase 6: Security & Flow Enhancements
Implement additional security measures.

**Requirements**:
- State parameter verification to prevent CSRF attacks
- Secure cookie handling for refresh tokens
- Proper error handling and user feedback
- Frontend integration considerations (redirect vs popup flow)

### Phase 7: Testing & Validation
**Post-implementation tasks**:
- Test complete OAuth flow
- Verify token generation and cookie setting
- Test auto-registration for new Google users
- Test login for existing Google users
- Run lint and typecheck
- Test with actual Google OAuth credentials

## Security Considerations
- Use HTTPS in production for OAuth redirects
- Validate state parameter to prevent CSRF
- Never expose client secret in frontend code
- Handle OAuth errors gracefully without revealing sensitive information
- Ensure proper token rotation and session management

## Frontend Integration Notes
- OAuth flow typically involves redirects or popup windows
- After successful OAuth, frontend should handle the returned access token
- Consider implementing both redirect and popup flows for flexibility
- Update frontend login UI to include "Login with Google" option

## Dependencies & Prerequisites
- Google Cloud Console project with OAuth 2.0 credentials configured
- Authorized redirect URIs matching the callback URL
- Proper scopes configured (email, profile)

## Risk Assessment
- **Low Risk**: Implementation follows existing patterns in the codebase
- **Medium Risk**: OAuth flow complexity and state management
- **Low Risk**: Database schema already supports OAuth users
- **Mitigation**: Thorough testing of OAuth flow and error handling

## Success Criteria
- Users can successfully authenticate via Google OAuth
- New Google users are auto-registered with verified emails
- Existing Google users can login seamlessly
- Hybrid token strategy works correctly for OAuth users
- Security measures (state verification, HTTPS) are properly implemented
- No breaking changes to existing local authentication flow
